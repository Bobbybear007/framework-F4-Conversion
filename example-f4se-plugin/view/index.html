window.prisma — injected automatically by PrismaUI_F4
PrismaUI_F4 injects window.prisma on every view before your OnDomReadyCallback fires. No C++ required in your plugin. Read and write Papyrus Auto properties and TESGlobal values directly from JavaScript. Reads return Promises; writes are fire-and-forget.

TESGlobal — read / write a global variable form
const val = await prisma.getGlobal(esp, formId) → number | null
prisma.setGlobal(esp, formId, value) → fire-and-forget float write

formId is the local hex ID without the file-index byte. A form at 0x00000800 in MyMod.esp is written as "800". ESL forms use the same convention — PrismaUI_F4 resolves the ESL prefix automatically via TESDataHandler::LookupForm.

Papyrus Auto Property — read / write a script property on any form
const val = await prisma.getProperty(esp, formId, scriptName, propName)
prisma.setProperty(esp, formId, scriptName, propName, value)

The form must have scriptName attached with the property declared Auto. Quest forms are the most reliable host — they persist across cell changes. Supports float, int, bool only (no strings, no arrays). Writes coerce your value to the property's declared type automatically. Always call after kPostLoadGame — the Papyrus VM is not ready before that.

null return — when and why
Both read functions return null (never throw) when:
• Plugin not in the active load order
• Form ID does not exist in that plugin
• Form exists but wrong type (not TESGlobal, or script not attached)
• Property not declared Auto or name mismatch
• Papyrus VM not yet ready (called before kPostLoadGame)

Always guard: if (val === null) { /* missing or not loaded */ }
C++ → JS — Invoke() and InteropCall()
api->Invoke(view, "fnName(args)") — evaluates a JS expression. Higher overhead; use for one-shot calls like init and infrequent UI updates.
api->InteropCall(view, "fnName", jsonArg) — calls a named window function directly. Lower overhead; use for per-frame or repeated data pushes. Both auto-convert ANSI game strings to UTF-8.

JS → C++ — BindUIEvent()
api->BindUIEvent(view, "eventName", callback) exposes a named function on window that JS calls like any other function.
The callback fires on the game thread — RE:: access is safe directly inside it. No AddTask needed.

RegisterJSListener() — Ultralight-thread callback
An alternative to BindUIEvent whose callback fires on the Ultralight render thread instead of the game thread.
Use when you need low-latency response or direct JSCore access. For any RE:: access inside such a callback, marshal to the game thread:
F4SE::GetTaskInterface()->AddTask([](){ /* safe here */ });

Always register inside OnDomReadyCallback — not before DOM ready.
View lifecycle — call-timing rules
RequestPluginAPI<IVPrismaUI4>() → on kGameDataReady
CreateView() → on kPostLoadGame or kNewGame
RegisterJSListener() → inside OnDomReadyCallback only
Hide(view) → immediately after CreateView unless you want it visible
IsValid(view) → always check after a game reload before using a stored handle

Violating these timings causes null crashes or silent no-ops.
